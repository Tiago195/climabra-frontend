# API Climabra — Catálogo de Endpoints

Referência completa de todos os endpoints da API. Inclui rotas, autenticação, request/response bodies.

---

## 🚨 Mudança recente: agenda agora é por turnos (morning/afternoon/night)

A lógica de agendamento mudou de **horários específicos** para **turnos**. Em vez de o cliente escolher "10:00", agora ele escolhe "manhã/tarde/noite" e o provider define a capacidade de cada turno.

### Resumo das mudanças visíveis ao front

| Onde | Antes | Agora |
|------|-------|-------|
| `Availability` | Um registro por dia, com `startTime/endTime/slotDurationMinutes` | Até **3 registros** por dia (um por turno), com `shift, startTime, endTime, capacity` |
| `AvailabilityException` | `startTime/endTime` opcionais para bloqueio parcial | Array `shifts: List<String>` — vazio = dia inteiro, ou turnos específicos |
| `Appointment` | `scheduledAt: ISO datetime` | `scheduledDate: LocalDate` + `shift: String` |
| `GET /providers/{token}/availability/slots` | `{ slots: ["2026-06-01T08:00:00", ...] }` | `{ shifts: [{ shift, startTime, endTime, capacity, available, blocked }, ...] }` |
| Endpoint público de visita | aceita `scheduledAt` ISO datetime | aceita `scheduledDate` (LocalDate) + `shift` |
| Endpoint público de signup | aceita `scheduledAt` ISO datetime | aceita `scheduledDate` (LocalDate) + `shift` |
| `POST /appointments` | aceita `scheduledAt` ISO datetime offset | aceita `scheduledDate` + `shift` |

### Valores possíveis de `shift`

`"morning"` | `"afternoon"` | `"night"`

### Defaults aplicados ao provider novo (em `createDefaultBusinessDays`)
- Manhã: **08:00–12:00**, capacidade **3**
- Tarde: **13:00–18:00**, capacidade **5**
- Noite: **18:00–22:00**, capacidade **3**
- Para os 5 dias úteis (seg-sex)

O provider pode mudar tudo — horários, capacidade, ativar/desativar turno por dia.

## Convenções gerais

### Base URL
Todas as rotas são relativas ao host configurado da API (ex: `http://localhost:8080` em dev).

### Content-Type
- Endpoints JSON: `application/json`
- Endpoints de upload: `multipart/form-data`

### Autenticação
Endpoints marcados como **🔒 Auth** exigem o header:
```
Authorization: Bearer <jwt-token>
```

O JWT é obtido via `POST /providers/login`. Token inválido/ausente retorna **401 Unauthorized**:
```json
{ "error": "Token não fornecido" }
```
ou
```json
{ "error": "Token inválido ou expirado" }
```

### Status HTTP padrão

| Código | Significado |
|--------|-------------|
| `200 OK` | Operação bem sucedida (leitura, update simples, transição de estado) |
| `201 Created` | Recurso criado |
| `204 No Content` | Operação bem sucedida sem corpo (delete, complete, cancel) |
| `400 Bad Request` | Erro de negócio (`ServiceException`) |
| `401 Unauthorized` | Token JWT inválido ou ausente |
| `404 Not Found` | Recurso não encontrado (alguns casos específicos) |
| `422 Unprocessable Entity` | Erro de validação `@Valid` no body |

### Formato de erro

**Erros de negócio** (400/404):
```json
{
  "message": "Cliente não encontrado",
  "status": "BAD_REQUEST",
  "fields": null
}
```

**Erros de validação** (422):
```json
{
  "message": "Erro de validação",
  "status": "UNPROCESSABLE_ENTITY",
  "fields": {
    "email": "Email é obrigatório",
    "password": "Senha deve ter no mínimo 8 caracteres"
  }
}
```

Todas as mensagens são em **português brasileiro**.

# 📋 Endpoints

## 1. Provider

Gerenciamento de prestadores (registro, autenticação, atualização de dados).

### `POST /providers` — Registrar provider
**Auth**: ❌ público
**Status sucesso**: `201 Created`

Request body:
```json
{
  "email": "provider@example.com",   // obrigatório, email válido
  "password": "senha12345",          // obrigatório, min 8 caracteres
  "name": "Tiago Silva",             // opcional
  "phone": "11999999999",            // opcional
  "companyName": "Climabra"          // opcional
}
```

Response body (`ResponseProviderDTO`):
```json
{
  "id": "uuid",
  "email": "provider@example.com",
  "name": "Tiago Silva",
  "phone": "11999999999",
  "companyName": "Climabra",
  "status": "active",                // active | blocked | canceled
  "publicToken": "uuid",
  "createdAt": "2026-05-25T10:00:00",
  "updatedAt": "2026-05-25T10:00:00"
}
```

Erros:
- `400` — `"Email já está registrado"`
- `422` — campos inválidos

**Diferença pós-refactor**: nenhuma.

---

### `POST /providers/login` — Login
**Auth**: ❌ público
**Status sucesso**: `200 OK`

Request body:
```json
{
  "email": "provider@example.com",   // obrigatório, email válido
  "password": "senha12345"           // obrigatório, min 8 caracteres
}
```

Response body (`Map<String, Object>`):
```json
{
  "provider": { /* ResponseProviderDTO */ },
  "token": "eyJhbGciOiJIUzI1NiJ9..."
}
```

Erros:
- `400` — `"Provider não encontrado"` (email não cadastrado OU senha inválida — mesma mensagem por segurança)

**Diferença pós-refactor**: nenhuma.

---

### `PUT /providers/{id}` — Atualizar provider
**Auth**: ❌ público _(observação: rota não exige JWT atualmente)_
**Status sucesso**: `200 OK`

Path params: `id` (UUID)

Request body (campos opcionais — apenas os enviados são atualizados):
```json

  "name": "Tiago Atualizado",
  "phone": "11888888888",

}
```

Response body: `ResponseProviderDTO` atualizado.

- `400` — `"Provider não encontrado"`

**Diferença pós-refactor**: nenhuma.

## 3. Availability

Gestão da agenda do provider (dias da semana ativos, exceções e slots disponíveis).

### `GET /availability` — Listar disponibilidades do provider logado
**Auth**: 🔒 sim
**Status sucesso**: `200 OK`

Retorna **até 21 registros** (7 dias × 3 turnos). Cada um representa a configuração de um turno em um dia da semana.

Response body (`List<ResponseAvailabilityDTO>`):
```json
[
  {
    "id": "uuid",
    "dayOfWeek": 1,           // 0=Domingo, 1=Segunda, ..., 6=Sábado
    "shift": "morning",       // "morning" | "afternoon" | "night"
    "startTime": "08:00:00",
    "endTime": "12:00:00",
    "capacity": 3,            // quantas visitas o provider aceita nesse turno
    "isActive": true
  },
  {
    "id": "uuid",
    "dayOfWeek": 1,
    "shift": "afternoon",
    "startTime": "13:00:00",
    "endTime": "18:00:00",
    "capacity": 5,
    "isActive": true
  }
]
```

---

### `POST /availability` — Criar/atualizar configuração de um turno
**Auth**: 🔒 sim
**Status sucesso**: `200 OK` _(upsert por (dayOfWeek, shift) do provider)_

Para configurar manhã, tarde e noite de um dia, faça 3 chamadas (uma por turno).

Request body:
```json
{
  "dayOfWeek": 1,                    // obrigatório, 0–6
  "shift": "morning",                // obrigatório: "morning" | "afternoon" | "night"
  "startTime": "08:00:00",           // obrigatório
  "endTime": "12:00:00",             // obrigatório, > startTime
  "capacity": 3,                     // obrigatório, 1–50
  "isActive": true                   // obrigatório (false = turno desativado nesse dia)
}
```

Response body: `ResponseAvailabilityDTO` salvo.

Erros:
- `400` — `"Horário de início deve ser anterior ao horário de fim"`
- `422` — campos inválidos / shift fora de morning|afternoon|night

---

### `GET /availability/exceptions` — Listar exceções
**Auth**: 🔒 sim
**Status sucesso**: `200 OK`

Response body (`List<ResponseExceptionDTO>`):
```json
[
  {
    "id": "uuid",
    "startDate": "2026-12-24",
    "endDate": "2026-12-26",
    "shifts": [],                    // [] = dia inteiro bloqueado em todas as datas
    "reason": "Feriado"
  },
  {
    "id": "uuid",
    "startDate": "2026-07-01",
    "endDate": "2026-07-31",
    "shifts": ["morning"],           // apenas manhãs de julho bloqueadas
    "reason": "Férias parciais"
  }
]
```

---

### `POST /availability/exceptions` — Criar exceção
**Auth**: 🔒 sim
**Status sucesso**: `200 OK`

Request body:
```json
{
  "startDate": "2026-12-24",         // obrigatório
  "endDate": "2026-12-26",           // obrigatório, ≥ startDate
  "shifts": ["morning", "night"],    // opcional. Vazio/ausente = dia inteiro
  "reason": "Feriado"                // opcional, max 255
}
```

Comportamento:
- `shifts` vazio/null/ausente: **todas** as datas do intervalo ficam totalmente bloqueadas
- `shifts` com valores: apenas esses turnos ficam bloqueados nas datas do intervalo

Regras (retornam `400`):
- `startDate` ≤ `endDate`

Response body: `ResponseExceptionDTO` salvo.

---

### `DELETE /availability/exceptions/{id}` — Deletar exceção
**Auth**: 🔒 sim
**Status sucesso**: `204 No Content`

Erros:

**Diferença pós-refactor**: nenhuma.

### `GET /providers/{publicToken}/availability` — Info pública do provider para signup


---

### `GET /providers/{publicToken}/availability/slots?date=YYYY-MM-DD` — Turnos disponíveis num dia
**Auth**: ❌ público
**Status sucesso**: `200 OK`

Path params: `publicToken` (UUID)
Query params: `date` (formato `YYYY-MM-DD`)

Response body (`ResponseSignUpSlotsDTO`):
```json
{
  "shifts": [
    {
      "shift": "morning",
      "startTime": "08:00:00",
      "endTime": "12:00:00",
      "capacity": 3,
      "available": 2,         // capacity - agendamentos confirmados
      "blocked": false
    },
    {
      "shift": "afternoon",
      "startTime": "13:00:00",
      "endTime": "18:00:00",
      "capacity": 5,
      "available": 0,
      "blocked": true         // bloqueado por exceção
    },
    {
      "shift": "night",
      "startTime": "18:00:00",
      "endTime": "22:00:00",
      "capacity": 3,
      "available": 3,
      "blocked": false
    }
  ]
}
```

Comportamento:
- `shifts` vazio: dia da semana sem turnos ativos (provider não atende nesse dia)
- Cada turno traz: horário configurado, capacidade total, quantas vagas restantes e se está bloqueado por exceção
- Use `available > 0 && !blocked` para mostrar como "disponível" no front
- Apenas turnos **ativos** aparecem (se o provider desativou a noite na segunda, ela não vem)

### `POST /clients/providers/{publicToken}/clients/{id}/appointments` — Solicitar visita (pelo portal)
**Auth**: ❌ público
**Status sucesso**: `201 Created` _(sem body)_

Request body:
```json
{
  "equipmentIds": ["uuid"],          // opcional — equipamentos já existentes do cliente
  "equipmentType": "split",          // opcional — se preenchido, cria um novo equipamento
  "equipmentBrand": "LG",            // opcional
  "equipmentModel": "M1",            // opcional
  "equipmentLabel": "Sala",          // opcional
  "description": "Não esfria",       // obrigatório
  "photoUrls": ["https://..."],      // opcional
  "problemType": "ruido",            // opcional
  "scheduledDate": "2026-06-01",     // obrigatório, ISO LocalDate
  "shift": "morning"                 // obrigatório: "morning" | "afternoon" | "night"
}
```

Regras:
- Pelo menos um dos equipamentos (`equipmentIds` ou `equipmentType`) deve resultar em algum equipamento
- Se algum equipamento já tem uma visita `scheduled`, retorna `400` com mensagem indicando o conflito

Response: vazio.

Erros:
- `400` — `"Link inválido"` / `"Um dos equipamentos selecionados já possui uma visita agendada..."` / `"Cliente não encontrado"`
- `422` — `shift` fora de `morning|afternoon|night`

```json
{
```

### `PUT /appointments/{id}/complete` — Concluir agendamento
**Auth**: 🔒 sim
**Status sucesso**: `204 No Content`

Regras de validação:
- Pelo menos um laudo deve existir para a visita
- Todos os laudos devem estar com status `completed`

Erros:
- `400` — `"Crie um laudo para cada equipamento antes de concluir o agendamento"`
- `400` — `"Aguarde todos os laudos serem aprovados pelo cliente antes de concluir"`
- `400` — `"Agendamento não encontrado"`

**Diferença pós-refactor**: nenhuma _(antes retornava `ResponseEntity<Void>` com `noContent()`, agora `void` com `@ResponseStatus(NO_CONTENT)` — comportamento HTTP idêntico)_.

---

### `PUT /appointments/{id}/cancel` — Cancelar agendamento
**Auth**: 🔒 sim
**Status sucesso**: `204 No Content`

Comportamento: cancela o agendamento e **deleta todos os laudos e itens de laudo** associados a ele.

Erros:
- `400` — `"Agendamento não encontrado"`

**Diferença pós-refactor**: nenhuma.

### `POST /reports` — Criar laudo
**Auth**: 🔒 sim
**Status sucesso**: `201 Created`

Request body:
```json
{
  "equipmentId": "uuid",             // obrigatório
  "appointmentId": "uuid",           // opcional
  "diagnosis": "Compressor falho",   // opcional
  "items": [                         // obrigatório, pelo menos 1
    { "description": "Trocar capacitor" },
    { "description": "Limpeza geral" }
  ]
}
```

Response body: `ResponseReportDTO`.

Regras:
- Se `appointmentId` informado e já existe laudo para esse `equipmentId` + `appointmentId`, retorna `400`
- O `clientId` é deduzido automaticamente do `equipmentId`

Erros:
- `400` — `"Equipamento não encontrado"` / `"Já existe um laudo para este equipamento nesta visita"`
- `422` — `items` vazio, sem `equipmentId`, ou `description` em branco

**Diferença pós-refactor**: nenhuma.

# Notas finais

- **Validação de campos**: erros `@Valid` sempre retornam `422` com mapa `fields`. Use isso para destacar campos inválidos no formulário.
- **Erros de negócio**: sempre `400` com `message` em PT-BR descritiva. São seguras para exibir direto ao usuário.
- **Datas**: a API usa `LocalDate` para `scheduledDate` (formato `YYYY-MM-DD`, sem hora ou timezone).
- **Public tokens**: o `publicToken` do provider é UUID; o do report é uma string de 48 caracteres (gerada concatenando dois UUIDs).
- **Schema do banco**: subir o Postgres do zero via `docker compose up -d postgres` — o Hibernate (`ddl-auto: update` em dev) cria todas as tabelas na primeira subida da API.
